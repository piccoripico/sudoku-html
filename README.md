## Overview

This repository contains the source code, tests, build script, and GitHub Actions workflows for the browser-based Sudoku game.
The distributed artifact is a single HTML file generated as `dist/Sudoku.html`.

## Repository Structure

- [`src/`](./src): HTML, CSS, and JavaScript source files
- [`tests/`](./tests): Automated tests for puzzle generation and state handling
- [`scripts/build.mjs`](./scripts/build.mjs): Builds the split source into a single-file `dist/Sudoku.html`
- [`docs/`](./docs): Screenshots and end-user documentation
- [`.github/workflows/`](./.github/workflows): CI, Pages, and release automation

## Development

1. Run `npm test`.
2. Run `npm run build`.
3. Open `dist/Sudoku.html` locally if you want to try the built file.

## CI

The CI workflow at [`.github/workflows/ci.yml`](./.github/workflows/ci.yml) runs for pull requests targeting `main`.
You can also run it manually from the `Actions` tab with `workflow_dispatch`.

## Distribution

`dist/` is intentionally ignored by Git.
For local testing, build `dist/Sudoku.html` and open it directly in your browser.
For online distribution, GitHub Pages can publish the built app automatically from GitHub Actions.
For downloadable distribution, use the GitHub Release asset generated by the release workflow, or upload the locally built `dist/Sudoku.html` yourself.

## GitHub Pages

The repository includes a dedicated Pages workflow at [`.github/workflows/pages.yml`](./.github/workflows/pages.yml).
When changes are pushed to `main`, GitHub Actions runs tests, builds `dist/Sudoku.html`, copies it to `public/index.html`, and deploys that artifact to GitHub Pages.
That means pull requests are checked by CI before merge, and merged changes are published by the Pages workflow after they reach `main`.

To enable it in the repository settings:

1. Open `Settings -> Pages`.
2. In `Build and deployment`, set `Source` to `GitHub Actions`.
3. Push to `main` or run the `Pages` workflow manually from the `Actions` tab.

## Release

1. Create a release tag such as `Sudoku_v1.1` or `v1.1.0`.
2. Push the tag to GitHub.
3. GitHub Actions runs tests, builds `dist/Sudoku.html`, and attaches it to the GitHub Release automatically.
